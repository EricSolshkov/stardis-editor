# 01 — 数据模型

本文档定义 Task Runner 系统的核心数据结构：任务（Task）、任务队列（TaskQueue）及其与场景模型的引用关系。

---

## 1. 概览

Task Runner 的职责是将场景配置转化为可执行的计算任务。核心概念：

```
SceneModel (场景数据)
    │
    │  引用 (Camera名/Probe名/场景文件)
    ▼
TaskQueue (任务队列)
├── Task[0]: Stardis IR Render  ──→ QProcess
├── Task[1]: HTPP Image         ──→ QProcess (依赖 Task[0] 的 .ht 输出)
├── Task[2]: Stardis Probe      ──→ QProcess
└── ...
```

- **引用语义**：任务通过名称引用场景对象（IRCamera、Probe），不复制数据
- **求值时机**：队列启动时一次性解引用并锁定参数快照，运行期间不再跟踪场景变化

## 2. 枚举定义

### 2.1 TaskType — 任务程序类型

```python
class TaskType(Enum):
    STARDIS = "stardis"
    HTPP = "htpp"
```

### 2.2 ComputeMode — Stardis 计算模式

一个 Stardis 任务**只能选择一种**计算模式。

```python
class ComputeMode(Enum):
    PROBE_SOLVE   = "probe_solve"    # 探针求解 (-p/-P/-f)
    FIELD_SOLVE   = "field_solve"    # 场求解 (-m/-s/-S/-F)
    IR_RENDER     = "ir_render"      # 红外渲染 (-R)
```

| 模式 | CLI 参数 | 场景引用 | 产出 |
|------|---------|---------|------|
| `PROBE_SOLVE` | `-p x,y,z[,t1,t2]` / `-P x,y,z:side` / `-f x,y,z` | `Probe` 对象列表 | stdout 数值（`T SE N_fail N_total`） |
| `FIELD_SOLVE` | `-m name[,t1[,t2]]` / `-s surface[,t1[,t2]]` / `-S surface[,t1[,t2]]` / `-F surface[,t1[,t2]]` | Body 名 或输入 STL 文件 | stdout 数值（-m/-s/-F）/ stdout VTK POLYDATA ASCII（-S） |
| `IR_RENDER` | `-R spp=:img=:fov=:pos=:tgt=:up=` | `IRCamera` 对象 | `.ht` 文件 |

### 2.3 FieldSolveType — 场求解子类型

```python
class FieldSolveType(Enum):
    MEDIUM_TEMP      = "medium_temp"       # -m name[,t1[,t2]]         介质平均温度        → stdout: T SE N_fail N_total
    SURF_MEAN_TEMP   = "surf_mean_temp"    # -s surface[,t1[,t2]]      面平均温度（输入 STL）→ stdout: T SE N_fail N_total
    SURF_TEMP_MAP    = "surf_temp_map"     # -S surface[,t1[,t2]]      面温度映射（输入 STL）→ stdout: VTK POLYDATA ASCII（per-triangle 温度场）
    SURF_FLUX        = "surf_flux"         # -F surface[,t1[,t2]]      面通量（输入 STL）   → stdout: T SE conv_W conv_SE rad_W rad_SE imp_W imp_SE tot_W tot_SE N_fail N_total
```

### 2.4 HtppMode — HTPP 后处理模式

```python
class HtppMode(Enum):
    IMAGE = "image"    # -i exposure:white=...
    MAP   = "map"      # -m pixcpnt=:palette=:range=:gnuplot
```

### 2.5 InputSource — HTPP 输入来源

```python
@dataclass
class InputFromTask:
    """引用队列中某个 IR_RENDER 任务的输出 .ht 文件"""
    task_id: str      # 被引用任务的 uuid

@dataclass
class InputFromFile:
    """直接指定已有 .ht 文件路径"""
    file_path: str    # 相对于 working_dir 的相对路径

InputSource = Union[InputFromTask, InputFromFile]
```

### 2.6 ErrorPolicy — 错误处理策略

```python
class ErrorAction(Enum):
    CANCEL   = "cancel"    # 取消整个队列
    SKIP     = "skip"      # 跳过当前任务继续
    PAUSE    = "pause"     # 暂停队列，等待用户决定

@dataclass
class ErrorPolicy:
    retry_count: int = 0                          # 重试次数，0 = 不重试
    after_retries_exhausted: ErrorAction = ErrorAction.CANCEL  # 重试耗尽后
```

## 3. Task — 单个任务

```python
@dataclass
class Task:
    # ── 标识 ──
    id: str                                    # uuid4
    name: str                                  # 用户可见名称
    task_type: TaskType                        # STARDIS | HTPP
    enabled: bool = True                       # 队列中可单独禁用

    # ── 执行环境 ──
    exe_ref: str = ""                          # 可执行文件引用（标签名 或 绝对路径）
    working_dir: str = ""                      # 工作目录（默认场景目录）
    env_vars: Dict[str, str] = field(default_factory=dict)

    # ── Stardis 专有 ──
    compute_mode: Optional[ComputeMode] = None
    stardis_params: Optional[StardisParams] = None

    # ── HTPP 专有 ──
    htpp_mode: Optional[HtppMode] = None
    htpp_params: Optional[HtppParams] = None
    input_source: Optional[InputSource] = None

    # ── 输出 ──
    output_redirect: Optional[str] = None      # 输出重定向文件名（相对 working_dir 的相对路径）
```

### 3.1 路径与可执行文件引用约定

#### exe_ref — 可执行文件引用

`exe_ref` 字段支持两种格式：

| 格式 | 示例 | 说明 |
|------|------|------|
| **标签名** | `"stardis-v3"` | 引用 `EditorPreferences.exe_tags` 中的标签，运行时解析为绝对路径 |
| **绝对路径**（回退） | `"D:/tools/stardis.exe"` | 当用户未配置标签时直接使用绝对路径 |

**判定规则**：若字符串包含路径分隔符（`/` 或 `\`）则视为绝对路径，否则视为标签名。

优先使用标签的好处：
- 可执行文件路径在 `EditorPreferences` 中集中管理，项目间共享
- 迁移机器时只需更新一处
- 避免项目文件中出现机器相关的绝对路径

#### 过程/结果文件路径

`output_redirect`、`InputFromFile.file_path`、`HtppParams.output_file` 等引用 `.ht`/`.ppm` 等过程或结果文件的路径字段，一律使用**相对于 `working_dir` 的相对路径**（通常为纯文件名）。

- 序列化时写入相对路径
- 运行时 `CommandBuilder` 直接传给子进程（QProcess 的 `workingDirectory` 已设为 `working_dir`）
- 若用户手动填写绝对路径，UI 应尝试转为相对路径；无法转换时保留原值并警告

## 4. Stardis 参数结构

### 4.1 StardisParams — 顶层

```python
@dataclass
class StardisParams:
    # ── 基本 ──
    model_file: str = ""                       # -M（通常由 SceneWriter 输出的 scene.txt 自动填充）
    samples: int = 1000000                     # -n（IR_RENDER 时无效，SPP 由相机控制）
    threads: int = 4                           # -t
    verbosity: int = 1                         # -V

    # ── 计算模式专有 ──
    probe_refs: List[str] = field(default_factory=list)       # PROBE_SOLVE: Probe 名称列表
    camera_ref: Optional[str] = None                          # IR_RENDER: IRCamera 名称
    field_solve: Optional[FieldSolveConfig] = None            # FIELD_SOLVE: 子类型配置

    # ── 高级选项 ──
    advanced: AdvancedOptions = field(default_factory=AdvancedOptions)
```

### 4.2 FieldSolveConfig

```python
@dataclass
class FieldSolveConfig:
    solve_type: FieldSolveType = FieldSolveType.MEDIUM_TEMP
    medium_name: str = ""              # -m: 介质名称
    surface_stl: str = ""              # -s/-S/-F: 输入 STL 文件路径（定义计算表面，必须是模型几何的子集）
    time_start: Optional[float] = None # 可选起始时间（秒），None 表示不传
    time_end: Optional[float] = None   # 可选结束时间（秒），None 表示不传
```

**时间范围语义**：
- 两者均为 `None`：不传时间参数（使用 stardis 默认值 = 无穷）
- 仅 `time_start` 非 `None`：传 `-S surface.stl,t1` 或 `-m name,t1`
- 两者均非 `None`：传 `-S surface.stl,t1,t2`，要求 `t1 ≤ t2`

**注意**：`-s/-S/-F` 的参数是**输入** STL 文件，定义计算表面（front sides of triangles）。计算结果输出到 stdout，需通过 `output_redirect` 捕获。

#### stdout 输出格式（来自 stardis 0.12 C 源码 `stardis-output.c`）

| FieldSolveType | CLI | stdout 格式 |
|----------------|-----|------------|
| `MEDIUM_TEMP` | `-m` | `{T_E} {T_SE} {N_fail} {N_total}\n`（T 均值、标准误，失败/总采样数，4 值） |
| `SURF_MEAN_TEMP` | `-s` | `{T_E} {T_SE} {N_fail} {N_total}\n`（同上，T 单位 K） |
| `SURF_FLUX` | `-F` | `{T_E} {T_SE} {conv_W} {conv_SE} {rad_W} {rad_SE} {imp_W} {imp_SE} {tot_W} {tot_SE} {N_fail} {N_total}\n`（T + 4 通量分量 W + 各自标准误 + 失败/总采样数，12 值；通量已乘以计算面积，单位 W） |
| `SURF_TEMP_MAP` | `-S` | VTK POLYDATA ASCII，CELL_DATA 含 4 个 SCALARS：`temperature_estimate`（float）、`temperature_std_dev`（float）、`failures_count`（int）、`computation_time_estimate` + `computation_time_std_dev`（float） |

**启用 `-e`（AdvancedOptions.extended_results）时**：所有模式改为人类可读文本格式，如 `"Temperature at boundary 'name' at t=X = T K +/- SE"`。

### 4.3 AdvancedOptions

```python
@dataclass
class AdvancedOptions:
```

## 5. HTPP 参数结构

### 5.1 HtppParams — Htpp 参数容器

```python
@dataclass
class HtppParams:
    # ── 通用 ──
    threads: int = 4                   # -t（默认 4 线程）
    force_overwrite: bool = False      # -f
    verbose: bool = False              # -v
    output_file: str = ""              # -o（空字符串 = 自动命名）

    # ── IMAGE 模式参数 (-i) ──
    exposure: float = 1.0              # 曝光度 [0, ∞)，默认 1.0
    white_scale: Optional[float] = None  # 白色缩放：None=自动，float=手动

    # ── MAP 模式参数 (-m) ──
    pixel_component: int = 0           # 像素分量 [0, 7]（默认 0=X）
    palette: str = ""                  # 调色板名称（空字符串 = "inferno" 默认）
    range_min: Optional[float] = None  # 范围最小值：None=自动，float=手动
    range_max: Optional[float] = None  # 范围最大值：None=自动，float=手动
    gnuplot: bool = False              # Gnuplot 输出标志
```

### 5.2 与 htpp C 源码的对齐

当前 `HtppParams` 完全对应 htpp C 实现（htpp 0.5）的参数体系。主要对应关系：

| Python | C 源码 | 说明 |
|--------|--------|------|
| `exposure` | `exposure`（默认 1.0） | 曝光度倍数 |
| `white_scale = None` | `white = -1.0` | 自动白点 |
| `white_scale = X` | `white = X` | 手动白点 |
| `pixel_component` | `pixcpnt` (0-7) | X/Y/Z + 不确定度，或时间估计 |
| `palette = ""` | `&scmap_palette_inferno` | 48 种预设调色板之一（scmap 库全部） |
| `range_min/max = None` | `{DBL_MAX, -DBL_MAX}` | 自动范围 |
| `range_min/max = X/Y` | `{X, Y}` | 手动范围 |

### 5.3 命令行生成细节

**仅生成非默认值原则**：命令生成时，若参数值等于默认值，则不传递该参数。

例如：
- 若 `exposure = 1.0`（默认），不生成 `-i` 标志
- 若 `exposure = 2.0`（非默认），生成 `-i exposure=2.0`
- 若 `white_scale = None`（默认自动），不在 `-i` 中添加 `white=...`
- 若 `white_scale = 5000`（手动），生成 `-i white=5000` 或 `-i exposure=1.0:white=5000`

完整规则见 [03_执行引擎.md §2.3 HTPP 命令构建规则](03_执行引擎.md#23-htpp-命令构建规则)。

## 6. TaskQueue — 任务队列

```python
@dataclass
class TaskQueue:
    tasks: List[Task] = field(default_factory=list)
    error_policy: ErrorPolicy = field(default_factory=ErrorPolicy)
    env_vars: Dict[str, str] = field(default_factory=dict)  # 队列级环境变量
```

`TaskQueue` 是 `SceneModel` 的一部分，随项目持久化。

### 6.0 环境变量继承

队列和任务各自维护一组环境变量（`Dict[str, str]`）。运行时按以下优先级合并：

```
系统环境变量  ←  队列 env_vars 覆盖/追加  ←  任务 env_vars 覆盖/追加
```

即：任务级变量 > 队列级变量 > 系统变量。

### 6.1 队列约束（编辑时校验 + 运行前校验）

| 约束 | 描述 | 触发时机 |
|------|------|---------|
| **HTPP 依赖** | `InputFromTask(id)` 引用的任务必须存在、类型为 `STARDIS`、模式为 `IR_RENDER`、且位于此任务之前 | 编辑时警告 + 运行前阻断 |
| **Camera 引用** | `IR_RENDER` 任务的 `camera_ref` 必须在 `SceneModel.cameras` 中存在 | 运行前阻断 |
| **Probe 引用** | `PROBE_SOLVE` 任务的 `probe_refs` 至少含一个有效 Probe 名 | 运行前阻断 |
| **exe 路径** | 任务的 `exe_path` 必须存在且可执行 | 运行前阻断 |
| **model_file** | Stardis 任务的 `stardis_params.model_file` 非空 | 运行前阻断 |
| **启用状态** | `enabled=False` 的任务直接跳过 | 运行时 |

### 6.2 错误处理流程

```
任务失败
  │
  ├── retry_count > 0 ?
  │     ├── 是 → 重试（递减 retry_count）
  │     │         └── 仍失败 → after_retries_exhausted
  │     └── 否 → after_retries_exhausted
  │
  └── after_retries_exhausted:
        ├── CANCEL → 终止队列，标记后续任务为"已取消"
        ├── SKIP   → 标记当前任务"已跳过"，继续下一个
        │             └── 依赖此任务的 HTPP 也标记"已跳过"
        └── PAUSE  → 暂停队列，弹出对话框让用户选择：
                      ├── 重试当前任务
                      ├── 跳过当前任务 → 继续
                      └── 取消队列
```

## 7. 场景引用解析

### 7.1 解引用时机

队列启动时，`TaskRunner` 对每个 `Task` 执行一次**快照求值**：

```python
def resolve_task(task: Task, model: SceneModel) -> ResolvedTask:
    """将引用转换为具体参数值。"""
    if task.compute_mode == ComputeMode.IR_RENDER:
        camera = model.get_camera_by_name(task.stardis_params.camera_ref)
        # → 冻结 camera.position, target, up, fov, spp, resolution
    elif task.compute_mode == ComputeMode.PROBE_SOLVE:
        probes = [model.get_probe_by_name(n) for n in task.stardis_params.probe_refs]
        # → 冻结每个 probe 的 position, type, side 等
```

### 7.2 引用失效处理

| 场景 | 处理 |
|------|------|
| 用户删除被引用的 Camera/Probe | 任务显示⚠️警告图标，运行前阻断 |
| 用户重命名 Camera/Probe | 由 SceneEditor 负责级联更新任务中的引用名 |

## 8. 与现有模型的关系

```
SceneModel (scene_model.py)
├── bodies, connections, probes, cameras, lights  ← 现有
├── task_queue: TaskQueue                          ← 新增
│   └── tasks: List[Task]
└── save_project() / load_project()               ← 扩展，增加 tasks 序列化
```

`TaskQueue` 不参与 `scene.txt` 的读写（解析器/写出器不变），仅在 `.stardis_project.json` 中持久化。

## 9. Stardis 验证模式（非任务）

以下为 stardis 的调试/验证功能，**不**建模为 Task，通过独立 UI 入口触发：

### 9.1 场景检查 (-d) — 已实现入口

**用途**：将场景文件加载为 stardis 内部模型并输出解析结果，用于验证场景定义正确性（几何体数量、边界条件匹配、材质参数等）。

**入口**：场景树任意位置右键菜单 → "验证场景…" / 场景菜单 → "验证场景"

**调用方式**：
```bash
stardis -M scene.txt -d dump_model.txt
```
读取 `dump_model.txt` 内容并在对话框中展示给用户（或嵌入日志面板）。

### 9.2 其他验证模式 — 仅文档记录，暂不实现

| 模式 | CLI | 推测用途 | 典型产出 |
|------|-----|---------|---------|
| **Dump C chunks** | `-c prefix` | 导出几何体离散化后的 C 源码块，用于嵌入式部署或调试网格质量 | `prefix_*.c` 文件 |
| **Dump 路径** | `-D type,file` | 导出蒙特卡洛光线追踪路径，用于可视化调试光线传播行为 | 路径数据文件（type 控制格式：`ascii`/`binary`） |
| **Green ASCII** | `-g` | 将 Green 函数矩阵以 ASCII 格式输出到 stdout，用于验证矩阵对称性和稀疏性 | stdout 文本 |
| **Green 二进制** | `-G file[,end_paths]` | 将 Green 函数矩阵导出为二进制文件，可选附带终止路径数。用于外部后处理或对比分析 | `.bin` 文件 |
